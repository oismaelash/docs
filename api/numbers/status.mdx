The Number Status endpoint returns the real-time connection state of a single WhatsApp number in your project. Use it to verify that a number is live before sending a critical message, to surface connection issues in your own dashboard, or to trigger a reconnection flow when a number goes offline.

`GET https://api.pilotstatus.com.br/v1/numbers/{id}/status`

## Path parameters

<ParamField path="id" type="string" required>
  The unique identifier of the number you want to check. Obtain this value from [GET /numbers](/api/numbers/list).
</ParamField>

## Example

```bash cURL
curl --request GET \
  --url https://api.pilotstatus.com.br/v1/numbers/num_abc123/status \
  --header 'Authorization: Bearer YOUR_API_KEY'
```

## Response

A successful request returns `200 OK` with the following fields.

<ResponseField name="id" type="string">
  The number identifier, matching the `id` you passed in the path.
</ResponseField>

<ResponseField name="status" type="string">
  Current connection state: `"connected"` or `"disconnected"`.
</ResponseField>

<ResponseField name="last_seen" type="string">
  ISO 8601 timestamp of the most recent activity recorded for this number (e.g., `"2024-01-15T09:45:00Z"`).
</ResponseField>

<ResponseField name="connection_type" type="string">
  How the number is connected: `"official"` for WhatsApp Business API or `"unofficial"` for a linked-device connection.
</ResponseField>

### Example response

```json
{
  "id": "num_abc123",
  "status": "connected",
  "last_seen": "2024-01-15T09:45:00Z",
  "connection_type": "unofficial"
}
```

## Handling disconnections

If `status` returns `"disconnected"`, restore the connection via the dashboard:

1. Navigate to **Dashboard → Numbers**, locate the number, and click **Reconnect**.

For `"unofficial"` numbers, reconnection requires the account holder to re-scan a QR code in the Pilot Status dashboard. The number will return to `"connected"` as soon as the scan is complete.

<Note>
  Poll this endpoint no more than once per minute. For instant disconnect alerts, subscribe to the `number.disconnected` webhook event instead — it fires the moment Pilot Status detects a lost connection, with no polling overhead.
</Note>
