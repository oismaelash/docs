# Check connection status

`GET /v1/numbers/{id}/status` returns the live connection state of one WhatsApp number.

```bash
curl "https://pilotstatus.com.br/v1/numbers/wa_abc/status" \
  -H "x-api-key: ps_your_token_here"
```

Response:

```json
{
  "id": "wa_abc",
  "state": "OPEN",
  "stale": false,
  "checkedAt": "2026-07-22T12:34:56.000Z"
}
```

<Note>
  This endpoint reflects the same connection state shown on the **Numbers** page in the dashboard (as does `GET /v1/numbers/{id}/connect`).
</Note>

## When to use it

- `GET /v1/numbers` does not force a mass refresh of connection state — call this endpoint per number when you need an up-to-date check.
- Poll it after showing a QR code / pairing code to detect when the customer completes the connection (or rely on the `number.connected` webhook instead).

## States

| `state` | Meaning |
| --- | --- |
| `OPEN` | Connected to WhatsApp. |
| `CLOSE` | Not connected — generate a new QR with `GET /v1/numbers/{id}/connect`. A pairing still in progress also reads as `CLOSE`; only a completed connection flips to `OPEN`. |

Treat `state === "OPEN"` as the definitive "ready to send" flag.

## How fresh is the answer: `stale`, `checkedAt`, `lastKnownAt`

Every `200` carries `stale`, plus **exactly one** of the two timestamps:

| Field | Type | Present when |
| --- | --- | --- |
| `stale` | boolean | Always. |
| `checkedAt` | string (ISO 8601) | `stale: false` — the provider answered this poll, so `state` is a live reading taken at this instant. |
| `lastKnownAt` | string (ISO 8601) | `stale: true` — the provider did **not** answer this poll, so `state` is the value we had stored, last updated at this instant. |

`stale: true` always comes with `state: "OPEN"`. The remembered value is only served when the stored state was already `OPEN`; for a number that was not `OPEN`, a provider that does not answer produces a `503` instead (see below).

**Meta (official Cloud API) numbers** answer `{ "id": "…", "state": "OPEN", "stale": false }` and carry **neither** timestamp — they are remote-managed, nothing is probed for them, so there is no reading to date.

## Errors

| Status | Body | Meaning |
| --- | --- | --- |
| `404` | `{ "error": "…", "state": null }` | No such number for this API key. **No `code` field.** |
| `503` | `{ "error": "…", "code": "…", "state": "…" }` | The connection check failed. `state` carries the last state stored for the number. |

The `code` on a `503` tells you whether retrying is worth it:

| `code` | What to do |
| --- | --- |
| `PROVIDER_NOT_CONFIGURED` | **Permanent.** Stop polling and raise it to an operator — retrying will not fix it. |
| `UPSTREAM_TIMEOUT` | Transient. Back off and retry. |
| `UPSTREAM_ERROR` | Transient. Back off and retry. |

<Note>
  **Fixed:** this endpoint used to answer `503` "WhatsApp provider not configured" for **every** number that was not already `OPEN` whenever the platform's own number was disconnected. That coupling is gone — a `503` now describes this number's own provider.
</Note>
