The List Numbers endpoint returns every WhatsApp number currently connected to your Pilot Status project. Use this endpoint to discover available `number_id` values for sending messages, to audit your connected numbers, or to check whether a particular number is active before initiating a send.

`GET https://api.pilotstatus.com.br/v1/numbers`

This endpoint takes no request parameters.

## Example

```bash cURL
curl --request GET \
  --url https://api.pilotstatus.com.br/v1/numbers \
  --header 'Authorization: Bearer YOUR_API_KEY'
```

## Response

A successful request returns `200 OK` with a `numbers` array. Each element contains the following fields.

<ResponseField name="id" type="string">
  The unique number identifier. Pass this value as `number_id` in all message-sending endpoints.
</ResponseField>

<ResponseField name="phone_number" type="string">
  The WhatsApp phone number including the country code, without a leading `+` (e.g., `5511999999999`).
</ResponseField>

<ResponseField name="display_name" type="string">
  The friendly name assigned to this number in your Pilot Status dashboard.
</ResponseField>

<ResponseField name="connection_type" type="string">
  How the number is connected. `"official"` indicates a WhatsApp Business API (cloud API) connection; `"unofficial"` indicates a linked-device connection.
</ResponseField>

<ResponseField name="status" type="string">
  Current connection state. `"connected"` means the number is ready to send and receive messages; `"disconnected"` means it requires reconnection.
</ResponseField>

<ResponseField name="created_at" type="string">
  ISO 8601 timestamp of when the number was first connected to your project (e.g., `"2024-01-01T00:00:00Z"`).
</ResponseField>

### Example response

```json
{
  "numbers": [
    {
      "id": "num_abc123",
      "phone_number": "5511999999999",
      "display_name": "Customer Support",
      "connection_type": "unofficial",
      "status": "connected",
      "created_at": "2024-01-01T00:00:00Z"
    }
  ]
}
```

<Tip>
  Cache number IDs in your application — they don't change as long as the number remains connected. Calling this endpoint on every message send adds unnecessary latency and is not recommended for high-throughput use cases.
</Tip>
