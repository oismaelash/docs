The Pilot Status REST API gives you programmatic access to WhatsApp messaging infrastructure. You can send text messages, media, and interactive content, manage connected numbers, and configure webhooks — all through standard HTTP requests and JSON payloads. This page covers everything you need to start making requests: the base URL, request format, authentication, error handling, and rate limits.

## Base URL

All API requests are made to the following base URL. Every endpoint in this reference is relative to it.

```
https://api.pilotstatus.com.br/v1
```

<Info>
  The API is versioned via the `/v1` path prefix. When breaking changes are introduced, a new version path will be released and announced in advance.
</Info>

## Request Format

All requests to the Pilot Status API must be sent as JSON. Set the `Content-Type` header to `application/json` on every request that includes a body. All responses are also returned as JSON, including error responses.

```bash
curl https://api.pilotstatus.com.br/v1/messages/send \
  -X POST \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{ "number_id": "your_number_id", "to": "5511999999999", "text": "Hello!" }'
```

<Note>
  Phone numbers in the `to` field must include the country code **without** a leading `+`. For example, use `5511999999999` for a Brazilian number, not `+5511999999999`.
</Note>

## Authentication

Every request must include a valid API key passed as a Bearer token in the `Authorization` header. See the [Authentication](/api/authentication) page for full details on obtaining and rotating your key.

```
Authorization: Bearer YOUR_API_KEY
```

## HTTP Status Codes

The API uses standard HTTP status codes to indicate whether a request succeeded or failed. Your integration should handle each code appropriately.

| Code | Meaning |
|------|---------|
| `200` | **Success** — The request completed successfully. |
| `400` | **Bad Request** — One or more request parameters are invalid. |
| `401` | **Unauthorized** — The API key is missing or invalid. |
| `404` | **Not Found** — The requested resource does not exist. |
| `422` | **Unprocessable Entity** — The request was well-formed but failed validation. |
| `429` | **Too Many Requests** — You have exceeded the rate limit for this number. |
| `500` | **Internal Server Error** — An unexpected error occurred on the server. |

## Error Response Format

When a request fails, the API returns a structured error object so you can identify the problem programmatically. Every error response follows this shape:

```json
{
  "error": {
    "code": "INVALID_NUMBER",
    "message": "The phone number format is invalid. Use country code without +.",
    "status": 400
  }
}
```

| Field | Type | Description |
|-------|------|-------------|
| `code` | string | A machine-readable error code identifying the specific failure. |
| `message` | string | A human-readable description of what went wrong. |
| `status` | number | The HTTP status code associated with this error. |

## Rate Limits

Pilot Status enforces per-number rate limits to ensure reliable delivery across all connected WhatsApp numbers. If you exceed the allowed request rate, the API responds with `HTTP 429 Too Many Requests`.

When you receive a `429` response, check the `Retry-After` header — it tells you how many seconds to wait before retrying. Implement exponential backoff to avoid hitting the limit repeatedly.

```bash
# Example 429 response headers
HTTP/1.1 429 Too Many Requests
Retry-After: 5
Content-Type: application/json
```

<Tip>
  Space out high-volume sends by queuing messages and respecting the `Retry-After` header. Exponential backoff — starting at the suggested delay and doubling on each failure — keeps your integration resilient under load.
</Tip>
