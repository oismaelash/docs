# Send interactive buttons

There is no separate `/messages/interactive` endpoint. Buttons are extra fields (`buttons`, `header`, `footer`) on the **single** send endpoint:

```text
POST https://pilotstatus.com.br/v1/messages/send
```

## Where buttons are allowed

- **Template sends** (`templateId`) — `buttons` override the template's own buttons, and can be combined with template media (including `video` / `document`; Meta Cloud API may reject some button \+ video/document combinations at delivery time).
- **Free-form text sends** (`text`) — `buttons` alongside the text; `header` and `footer` require `buttons`.
- **Not allowed** in plain direct-media mode (`media` \+ `mediaType` without `templateId`/`text`) — see [Send a Message](/api/messages/send).

Up to **3** buttons per message. Types: `reply`, `url`, `call`, `copy`.

## Example: template \+ buttons

```bash
curl -X POST "https://pilotstatus.com.br/v1/messages/send" \
  -H "Content-Type: application/json" \
  -H "x-api-key: ps_your_key_here" \
  -d '{
    "templateId": "confirm-template",
    "destinationNumber": "+5511999999999",
    "variables": { "name": "John" },
    "buttons": [
      { "type": "reply", "displayText": "Confirm", "id": "confirm" },
      { "type": "url", "displayText": "View details", "url": "https://example.com/order/123" }
    ]
  }'
```

## Carousel

A **session carousel** attaches a horizontal, swipeable row of **2–10 cards** to a **free-form** send — no template and no approval. It is a `carousel` object plus a message-level `text` that becomes the **carousel body**. Each card carries its own **image**, an optional **body**, and its own **quick-reply buttons**. Session carousels work on **both** number types — Meta Cloud API and unofficial (Pilot Status web); on **Meta** numbers, though, they only send inside the **24-hour service window** — outside it, Meta rejects free-form, so use an approved **template carousel** ([Templates](/api/templates)) instead:

```json
{
  "text": "Check out our latest offers",
  "destinationNumber": "+5511999999999",
  "carousel": {
    "cardTemplate": { "hasBody": true, "buttons": [{ "type": "QUICK_REPLY" }] },
    "cards": [
      { "media": { "format": "IMAGE", "url": "https://media.example.com/a.png" },
        "body": "Sneakers — 30% off",
        "buttons": [{ "type": "QUICK_REPLY", "text": "I want it", "payload": "SKU-1" }] },
      { "media": { "format": "IMAGE", "url": "https://media.example.com/b.png" },
        "body": "Backpack — free shipping",
        "buttons": [{ "type": "QUICK_REPLY", "text": "I want it", "payload": "SKU-2" }] }
    ]
  }
}
```

<Warning>
  On **Meta** numbers a **URL button in a session carousel is impossible** — Meta returns `400`; Meta session-carousel buttons are **`QUICK_REPLY` only**, and a `QUICK_REPLY` \+ `URL` mix in one card is allowed **only** in a *template* carousel ([Templates](/api/templates)). On **unofficial (Pilot Status web)** numbers a **`URL`** button works ad-hoc — it maps to a native link button.
</Warning>

- `cardTemplate` is the **shared skeleton** every card must match — one `hasBody` decision and one button list (**types and order**). Each card only fills in its own `media`, `body`, and button **values** (`text`, `payload`).
- **`hasBody` is all-or-nothing**: if `cardTemplate.hasBody` is `true`, every card needs a `body`; if `false`, none may. Each card's `buttons` must match `cardTemplate.buttons` by **count, type and order** (the skeleton's types — `QUICK_REPLY` on Meta, or `URL` on unofficial numbers).
- Caps (enforced — `422`/`400` on violation): **2–10 cards**; card `body` up to **160** chars (all cards or none); up to **2 buttons per card**; button `text` (label) up to **20** chars; button `payload` (id) up to **256** chars; message-level **`text` required**, up to **1024** chars (the carousel body).
- **Media rule**: `media.format` is `IMAGE`; `media.url` must be a **public `https` URL ending in `.png`, `.jpg`, `.jpeg`, or `.webp` with no query string**. An unfetchable URL is refused **before** dispatch with a typed error naming the card — Meta would otherwise accept it with `200` and then deliver nothing.
- **Mutual exclusion**: `carousel` requires `text` and **cannot** be combined with `templateId`, `list`, `buttons`, or a direct media send (`media` \+ `mediaType`); a `header` is **not** allowed (the media lives on each card).
- A card's quick-reply tap arrives as a **normal inbound webhook**, exactly like a reply button — see [How button replies arrive](#how-button-replies-arrive) below.

```bash
curl -X POST "https://pilotstatus.com.br/v1/messages/send" \
  -H "Content-Type: application/json" \
  -H "x-api-key: ps_your_key_here" \
  -d '{
    "text": "Check out our latest offers",
    "destinationNumber": "+5511999999999",
    "carousel": {
      "cardTemplate": { "hasBody": true, "buttons": [{ "type": "QUICK_REPLY" }] },
      "cards": [
        { "media": { "format": "IMAGE", "url": "https://media.example.com/a.png" },
          "body": "Sneakers — 30% off",
          "buttons": [{ "type": "QUICK_REPLY", "text": "I want it", "payload": "SKU-1" }] },
        { "media": { "format": "IMAGE", "url": "https://media.example.com/b.png" },
          "body": "Backpack — free shipping",
          "buttons": [{ "type": "QUICK_REPLY", "text": "I want it", "payload": "SKU-2" }] }
      ]
    }
  }'
```

## Response (202)

```json
{
  "id": "msg_abc",
  "correlationId": "corr_123",
  "status": "QUEUED"
}
```

## How button replies arrive

When the contact taps a **reply** button, you receive a **normal inbound webhook** — `message.reply` (or `message.received`) — whose `content` is the **button label** (e.g. `"Confirm"`), with `quotedMessageId` pointing at your original message.

<Warning>
  There is **no** `button_reply` event or message type and **no** `button_id` field — the `id` you set on a reply button is not echoed back. Match button taps by the label in `content` plus `quotedMessageId` (and `correlationId` when present).
</Warning>

See the full payloads in [Webhook events](/api/webhooks/events) and the full parameter list (button object shapes, `header`/`footer` rules) in [Send a Message](/api/messages/send).
