Pilot Status delivers webhook events to any HTTPS endpoint you register. You can register endpoints through the dashboard for a quick setup or through the API when you need to automate webhook management as part of your deployment pipeline. Both methods support filtering by event type and scoping by number.

---

## Configure via Dashboard

The dashboard is the fastest way to register a webhook endpoint without writing any code.

1. Open the **Pilot Status Dashboard** and navigate to **Settings**.
2. Select **Webhooks** from the settings menu.
3. Click **Add Endpoint**.
4. Enter your HTTPS endpoint URL in the **URL** field.
5. Select the events you want to receive, or leave all events selected to receive everything.
6. Click **Save**.

Your endpoint will start receiving events immediately after saving.

## POST /v1/webhooks

Use this endpoint to register a webhook programmatically. Pilot Status will begin delivering events to your URL as soon as the request succeeds.

**Endpoint:** `POST https://api.pilotstatus.com.br/v1/webhooks`

<ParamField body="url" type="string" required>
  Your HTTPS endpoint URL. Pilot Status will POST event payloads to this address. Must use `https://` — HTTP URLs are rejected.
</ParamField>

<ParamField body="events" type="array of strings">
  List of event type strings to subscribe to, e.g. `["message.received", "message.delivered"]`. Omit this field to subscribe to all events. See the [Events reference](/api/webhooks/events) for the full list of available event types.
</ParamField>

<ParamField body="number_id" type="string">
  Scope this webhook to a single number by providing its ID. Omit to receive events from all numbers on your account.
</ParamField>

```bash
curl -X POST https://api.pilotstatus.com.br/v1/webhooks \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://yourapp.com/webhook",
    "events": ["message.received", "message.delivered"]
  }'
```

## GET /v1/webhooks

Retrieve a list of all webhook endpoints registered on your account, including their IDs, subscribed events, and current status.

```bash
curl https://api.pilotstatus.com.br/v1/webhooks \
  -H "Authorization: Bearer YOUR_API_KEY"
```

The response returns an array of webhook objects. Each object includes the webhook `id` you need for update and delete operations.

---

## DELETE /v1/webhooks/{id}

Remove a registered webhook endpoint by its ID. Pilot Status will stop delivering events to that URL immediately.

```bash
curl -X DELETE https://api.pilotstatus.com.br/v1/webhooks/wh_abc123 \
  -H "Authorization: Bearer YOUR_API_KEY"
```

Replace `wh_abc123` with the ID returned when you created the webhook or retrieved from `GET /v1/webhooks`.

## Retry behavior

When your endpoint returns a non-2xx HTTP response — or does not respond within the timeout window — Pilot Status automatically retries delivery using exponential backoff:

| Attempt | Delay after previous failure |
|---------|------------------------------|
| 1st retry | 10 seconds |
| 2nd retry | 30 seconds |
| 3rd retry | 2 minutes |
| 4th retry | 10 minutes |

After **5 consecutive failures**, Pilot Status marks the webhook as **failing** and surfaces an alert in your dashboard. The webhook remains registered and will resume delivery once your endpoint recovers.

<Info>
  To manually re-enable a failing webhook after fixing your endpoint, open the Webhooks section of your dashboard and click **Re-enable** next to the affected endpoint.
</Info>

<Warning>
  Webhook URLs must use HTTPS. HTTP endpoints are not accepted for security reasons.
</Warning>
